Start Here · Part 1

What Everything Actually Is

Before any commands. Read this once. It will make everything else click.

🐙
Git

Git is a tool that tracks every change you make to your code over time. It lives inside your Codespace. It's like a very smart Ctrl+Z that never forgets anything and can sync with others.

☁️
GitHub

GitHub is a website that stores your project in the cloud so both teammates can access it. Think of it as the shared hard drive that lives on the internet. github.com/nveigacoutinho-ops/pharmabot is your project's address.

💻
Codespace

A Codespace is a computer that runs in your browser tab. It has your files, your editor, and your terminal. It's your personal workspace. Your teammate has their own separate one. Changes you make here do NOT automatically go to GitHub.

📦
Repository (Repo)

The repository is the project folder — both on GitHub (the cloud copy) and in your Codespace (your local copy). When people say 'the repo', they mean the whole project: all files, all history, all branches.

⚠️ Critical distinction: Your Codespace and GitHub are TWO DIFFERENT PLACES. You must manually send your work from your Codespace to GitHub every time. That's what git push does.

Local vs Remote — the most important concept

Local (your Codespace)

This is YOUR copy of the project. Only you can see it. Changes here are invisible to your teammate until you push. If your Codespace crashes or resets before you push, unsaved work can be lost. Location: inside your browser tab.

Remote (GitHub)

This is the SHARED copy on github.com. Both teammates can see it. This is the source of truth. This is what matters. This is what gets deployed. Always push here before closing your browser.

The difference between these four things

Action What it does Is work safe?
Typing code Changes appear in the editor ❌ Only in editor — gone if tab closes
Saving the file (Ctrl+S) File is saved in Codespace memory ⚠️ Safe in Codespace, not on GitHub
git commit Creates a snapshot in your local Git history ✅ Saved in Codespace git history
git push Uploads your commits to GitHub ✅✅ Safe on GitHub — teammate can see it
Part 2 · Critical

How to Actually Save Your Work — The Full Chain

This is the single most important thing to understand. Most beginner mistakes come from not knowing where in this chain their work actually is.

🖊️
Step 1 — You type code in the editor
You can see the changes on screen. The file has an unsaved dot next to its name in the tab. Nothing has been saved anywhere yet.
❌ Gone if tab closes
💾
Step 2 — You save the file — Ctrl+S (or Cmd+S on Mac)
The file is now saved inside your Codespace. The dot disappears from the tab name. Your teammate still cannot see this. GitHub still has the old version.
⚠️ Codespace only
🔍
Step 3 — You run git status
Git tells you which files have changed since your last commit. You should run this constantly. It's harmless — it's just checking.
⚠️ Just checking — nothing changes
📦
Step 4 — You stage changes — git add .
You're telling Git: 'these are the changes I want to include in my next commit'. Think of it as putting files in an envelope before sealing it. The dot (.) means 'everything changed'.
⚠️ Staged — still Codespace only
🏷️
Step 5 — You commit — git commit -m "message"
You seal the envelope and write a label on it. Git saves a permanent snapshot of your changes in the Codespace's local history. If Codespaces crashes now, this is preserved in your Codespace — but still NOT on GitHub.
✅ In local git history
🚀
Step 6 — You push — git push
The envelope is delivered to GitHub. Your commits are now on the cloud. Your teammate can see them. The project is backed up. This is the only step that truly saves your work permanently.
✅✅ On GitHub — safe and shared
✅ Rule of thumb: If you haven't pushed, it's not saved. Commit and push at least at the end of every working session — even if your work isn't finished.
⚠️ Before closing your browser or laptop: always run git add . && git commit -m 'WIP' && git push. 'WIP' = Work In Progress. It's better to push unfinished work than lose it.

Part 6 · Teamwork Foundation

Branches — The Most Important Concept for Teamwork

What is a branch?

A branch is an independent copy of the project that only you work on. Think of main as the official published book that everyone reads. Your branch is your personal draft — you write in your draft, and when it's ready and reviewed, it gets published into the official book.

main  ●───────────────────────────────● (stable, always works)
           \                         /
feature     ●────●────●────●────●──●   (your work, isolated)

Why does main need to stay stable?

main is the shared foundation. If both people work directly on main and one of you makes a mistake, the whole project breaks for both of you. Branches protect main by keeping all in-progress work separate until it's ready.

🚫 Rule: Never write code directly on main. Always create a branch first — even for tiny changes.

When do you create a branch?

🧩

One branch per task

Each distinct piece of work gets its own branch. One branch for the chatbot engine, another for the Euclid jam tree, another for the main menu. When the task is done and merged, the branch is deleted.

⚡

Keep branches small

A branch that does one clear thing is easy to review and merge. A branch that does 10 things is hard for your teammate to understand. Small branches → fast merges → fewer conflicts.

🗑️

Delete after merging

After a branch is merged into main, delete it. It has served its purpose. Keeping old branches around is clutter and confusion.

All branch commands you need

# CREATE a new branch (do this for every new task)
git checkout -b feature-chatbot-engine

# SEE all branches (* marks the one you're on)
git branch

# SWITCH to an existing branch
git checkout main
git checkout feature-chatbot-engine

# PUSH branch to GitHub for the first time
git push -u origin feature-chatbot-engine

# After first push, just use:
git push

# DELETE a branch after it's been merged
git branch -d feature-chatbot-engine

What does git push -u origin feature-chatbot-engine mean?

Part Meaning
git push Send my commits from here (Codespace) to GitHub
-u "Set upstream" — link this branch here to the same-named branch on GitHub. Only needed once per branch.
origin The nickname Git uses for your GitHub repo. It's automatically set when you open a Codespace from GitHub.
feature-chatbot-engine The name of the branch you're pushing.

After running git push -u origin branch-name once, future pushes from that branch need only git push — Git remembers the link.

⚠️ What happens if both people edit main directly? Person 1 pushes → fine. Person 2 tries to push → rejected because main has moved on. Person 2 must now pull, fix any conflicts, and push again. This is avoidable entirely by using branches.
Part 7 · Code Review

Pull Requests — More Than Just Merging

A Pull Request (PR) is not just a button you click to merge. It's the heart of collaborative coding. It's how two people can work on the same project without stepping on each other — even when working on the same file.

👁️

Review before merging

A PR shows your teammate exactly what you changed — every line, every file. They can read it before it goes into main. This catches mistakes before they become problems.

💬

Comment on specific lines

Your teammate can click any line in your PR and leave a comment: "this will break if the JSON is empty" or "rename this variable — it's confusing". This is how professionals review code.

🛡️

Protect main

A PR is a gate. Nothing gets into main without going through this gate. Even in a 2-person student project, this habit prevents broken code from reaching the shared copy.

📝

A written record

Every PR is a log entry: what was done, why, and who approved it. Three months later you can read the PR history and understand exactly how the project evolved.

How to create a PR — step by step

1

Go to the repo

Navigate to github.com/nveigacoutinho-ops/pharmabot

2

Look for the yellow banner

After pushing a branch, GitHub shows a yellow banner: "Compare & pull request". Click it. If it's gone: Pull Requests tab → New Pull Request.

3

Set the direction

base: main (where code will go INTO). compare: your-branch (where code is coming FROM). Always double-check this.

4

Write a clear title and description

Title: what you did. Description: why you did it and what to look out for. Example title: "Add chatbot question loop to chatbot.py". Your teammate will thank you.

5

Create and wait for review

Click Create Pull Request. Your teammate reviews, comments, and approves. Once approved: Merge pull request → Confirm merge. Then clean up.

# After merging:
git checkout main          # go back to main
git pull                   # get the merged code
git branch -d feature-chatbot-engine   # delete old branch
✅ Even in a 2-person project: always create a PR. Never merge your own branch directly. The review step — even a quick look — catches mistakes that you miss because you wrote the code.
Part 8 · How to Work Together

Two Ways to Collaborate Safely

There's no single right way to split work as beginners. Here are two models. Pick the one that fits your situation — or use both depending on the task.

Model A — Split by File

Each person owns specific files

The safest model for beginners. You and your teammate agree upfront: Person 1 owns all Python files, Person 2 owns all JSON files. You rarely touch the same file, so conflicts are nearly impossible.

Best for: when you have clearly separable areas of work (like PharmaBot — Python engine vs JSON knowledge base).

✅ Near-zero conflicts
✅ Easy to understand
✅ No coordination needed mid-task
⚠️ Less flexible — hard to help each other
Model B — Branch Per Task (Flexible)

Anyone can work on anything

Either teammate can work on any file, as long as you each have your own branch. This is how real teams work. The rule is: one task = one branch = one person at a time.

Best for: when work isn't neatly separable, or when you want to help each other on any file.

✅ Flexible — no file ownership
✅ Can help each other freely
⚠️ Requires communication
⚠️ More conflicts possible

Safe rules for Model B (flexible)

🗣️

Announce before starting

Tell your teammate: "I'm about to edit chatbot.py". If they're also editing it, coordinate. A quick message prevents an hour of conflict resolution.

⬇️

Always pull before starting

Run git pull every time you begin. Even if you worked an hour ago. Your teammate may have pushed since then.

🌿

Open PRs early

Push your branch and create a PR even before you're done. Label it "WIP" (Work In Progress). Your teammate can see what you're working on and avoid duplicating effort.

🔄

Keep branches short-lived

Don't let a branch sit open for 3 days. The longer a branch lives, the more it diverges from main, and the harder it is to merge. Aim: open a branch, finish the task, merge within the same session or day.

💡 For PharmaBot specifically: start with Model A (split by file). It's safer for beginners. Switch to Model B once you're comfortable with branches and PRs.

Part 11 · When Things Collide

Resolving Merge Conflicts

A conflict happens when two people edit the exact same line of the same file on different branches. Git can't decide which version to keep, so it puts both in the file and asks you to choose.

What a conflict looks like in the file

<<<<<<< HEAD
def start_chatbot():
    print("Welcome to PharmaBot")
=======
def start_chatbot():
    print("PharmaBot Ready")
>>>>>>> feature-chatbot-engine
<<<<<<< HEAD = what's in main. ======= separator. >>>>>>> = what's in your branch.

How to fix it — 3 steps

1

Delete all conflict markers

Remove the <<<<<<< HEAD, =======, and >>>>>>> lines. Keep only the code you want.

2

Edit the file to what it should be

Either keep one version, or combine both into something correct. Save the file.

3

Stage, commit, push

git add .
git commit -m "resolve conflict in chatbot.py"
git push

How to prevent most conflicts

🗂️

Stick to your assigned files

If you follow the team split, conflicts are nearly impossible.

⬇️

Pull every session

git pull at the start brings in what your teammate pushed. The longer you wait, the more divergence.

💬

Communicate

Before touching a file the other person may be editing, send a message. 10 seconds of communication prevents 20 minutes of conflict resolution.

Part 14 · When Things Feel Wrong

Before You Panic — The 5-Second Checklist

90% of beginner Git problems come from one of these six things. Before assuming something is broken, run through this list.

🌿
Am I on the right branch?
git branch
The * shows your current branch. Never work on main.
💾
Did I save the file?
Ctrl+S (or Cmd+S)
Unsaved files show a dot (●) in the editor tab.
📦
Did I stage and commit?
git status
Should say "nothing to commit" if you committed.
🚀
Did I push?
git status
If it says "ahead by N commits", you haven't pushed yet.
⬇️
Did I pull before starting?
git pull
Your teammate may have pushed since you last pulled.
📁
Am I in the right folder?
ls && pwd
You should see app/, decision_tree/, tests/, README.md

What beginners most often confuse

❌

Common misconceptions

"I saved the file so my teammate can see it." Wrong. Saving is only local. You must also commit and push.
"I committed so it's on GitHub." Wrong. Commit is local. Only push sends it to GitHub.
"I don't need to pull — I just opened my Codespace." Wrong. Codespace opens in the state it was last in. Your teammate may have pushed since then.
"git status changes my files." Wrong. git status is read-only. It never modifies anything.
✅

The correct mental model

Saving a file → only in your editor tab
git add + git commit → saved in Codespace git history
git push → saved on GitHub, visible to teammate
git pull → brings teammate's pushed work into your Codespace
🧭 Quick diagnostic — run this any time you're confused:
# Run all of these. Read the output carefully. pwd # where am I? ls # is the project here? git branch # which branch am I on? git status # what's the state of my changes? git log --oneline -5 # what were my last 5 commits?
Part 15 · Recovery Guide

Common Mistakes & How to Fix Them

You will encounter at least one of these. Every developer has. Bookmark this section.

😬 "I wrote code but GitHub doesn't show it"

This is the most common beginner mistake. Code only appears on GitHub after you push. Check where you are in the chain:

git status # shows what's uncommitted git add . git commit -m "description of what you did" git push # this is what sends it to GitHub
😬 "I saved the file but my teammate still can't see it"

Saving a file in the editor only saves it inside your Codespace — not on GitHub. Your teammate sees what's on GitHub. You must commit and push.

git add . && git commit -m "update" && git push
😬 "I forgot what branch I'm on"
git branch # the * marks your current branch git status # also shows branch name at the top
😬 "I accidentally wrote code on main"

Don't panic. Your work is not lost. Save it to a new branch, then clean up main:

# Step 1: move your work to a new branch (before committing anything) git checkout -b my-accidental-work # Step 2: commit and push it there git add . git commit -m "save accidental main work" git push -u origin my-accidental-work # Step 3: reset main back to what's on GitHub git checkout main git reset --hard origin/main
Your work is safe on the new branch. Main is clean again. Create a PR from the new branch as normal.
😬 "I committed but didn't push — did I lose my work?"

No. Your commit is saved in the Codespace's local git history. Just push:

git push

If the branch doesn't exist on GitHub yet, use: git push -u origin your-branch-name

😬 "My push was rejected"
! [rejected] main -> main (fetch first)

This means your teammate pushed since you last pulled. Your local history is behind GitHub. Fix:

git pull --rebase git push

If there are conflicts after rebasing, fix them (see the Conflicts section), then push again.

😬 "I pulled and now things look different / something is missing"

This is normal. Your teammate merged changes that you didn't have. git pull brought those in. Your code is still there — it's just been joined by your teammate's. Run:

git log --oneline -10 # see the last 10 commits — yours and theirs git diff HEAD~1 # see what changed in the last commit
😬 "I opened the terminal and don't know what folder I'm in"
pwd # Print Working Directory — shows your current location ls # list files here cd /workspaces/pharmabot # navigate directly to project root ls # should now see: app/ decision_tree/ tests/ README.md
😬 "My Codespace looks different from what's on GitHub"

Your Codespace may be on a different branch, or your teammate pushed while you had the Codespace open. Fix:

git checkout main # switch to main git pull # get the latest from GitHub git status # confirm you're in sync
😬 "I committed the wrong message"

If you haven't pushed yet — fix it:

git commit --amend -m "correct message here"
If you already pushed — leave it. Don't rewrite history on a shared repo. It causes more problems than a bad message.
😬 "I want to undo my last commit but keep the code changes"
git reset --soft HEAD~1

Your code changes stay exactly as they were. The commit disappears. You can re-commit with a better message or more files included.

😬 "I forgot to pull before coding — now I'm getting a conflict warning"
git pull --rebase # brings in latest and replays your commits on top

This usually works cleanly. If conflicts appear, fix them manually (see the Conflicts section), then run git rebase --continue, then git push.

Part 16 · Write for Future You

Commit Message Rules

This matters more than it seems. After 3 months you'll look back at the git history and either thank yourself or be completely lost. A commit message is a note to your future self and your teammate.

❌ Bad — Useless messages
  • update
  • fix
  • changes
  • stuff
  • asdfgh
  • wip2
  • final
✅ Good — Tells a story
  • add chatbot question loop in chatbot.py
  • fix tree_loader failing when JSON file is missing
  • write J001 euclid jam full decision tree
  • add sensor error option to main menu
  • rename start() to start_chatbot() for clarity
  • WIP: barcode sensor tree — questions done, resolutions todo
Formula: verb + what you changed + where (if not obvious)

Common verbs: add fix update remove rename refactor test WIP
💡 WIP commits are fine. If you need to stop mid-task, push a WIP commit: git commit -m "WIP: chatbot loop — answer handling not done yet". It's better than losing work or not pushing. You can clean up the message later.
