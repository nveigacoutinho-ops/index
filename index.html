Real commands — copy this pattern every day
# 1. Always start by getting the latest version git pull # 2. Write your code (in the editor) # 3. Stage everything you changed git add . # 4. Commit with a clear message git commit -m "added chatbot question loop" # 5. Push to GitHub git push
⚠️ Why this order matters: If you skip git pull at the start, you might be writing code based on an old version. Then when you push, Git sees a conflict between your old base and what's on GitHub. That's how you get "merge conflicts" — the most common beginner problem.
Parts 4 & 5

Branches — The Most Important Concept for Teamwork

What is a branch?

A branch is a copy of the project that only you work on. Changes you make on your branch don't affect the main project until you deliberately merge them.

📚 main branch = the official published book
✏️ your branch = your personal draft

You write in your draft. When it's ready and reviewed, it gets added to the official book.

Why branches exist for PharmaBot specifically

Imagine both of you edited chatbot.py at the same time without branches:

Person 1 saves version A Person 2 saves version B Person 1 pushes → GitHub has version A Person 2 pushes → GitHub gets confused: which is right?
✅ With branches: you each work in your own space. No collision.

Working with Branches — All the Commands

🌿
Create your branch
Do once per feature or task
git checkout -b your-branch-name
🔍
Check which branch you're on
The one with * is your current branch
git branch
🔀
Switch between branches
Go back and forth as needed
git checkout main git checkout feature-chatbot
🚫 Never code directly on main. Always create a branch first.

Real branch names for PharmaBot

Person 1 — Python
# Working on Python logic git checkout -b feature-chatbot-engine
Person 2 — JSON
# Working on decision trees git checkout -b feature-decision-trees-jams

Full workflow on a branch — step by step

# 1. Start your branch git checkout -b feature-chatbot-engine # 2. Pull latest (important even on a new branch) git pull # 3. Write your code... # 4. Save your work git add . git commit -m "built chatbot question loop" # 5. Push your branch to GitHub (first push) git push -u origin feature-chatbot-engine
The -u origin feature-chatbot-engine part is only needed the first time. After that, just git push works.

Pull Requests — How Code Gets Into Main

A Pull Request (PR) is how you say: "I finished something on my branch. Please review it and merge it into main."

How to create a PR

1
Go to the repo
Navigate to github.com/nveigacoutinho-ops/pharmabot
2
Click the banner
You'll see a yellow banner: "Compare & pull request" — click it.
If no banner: Pull Requests tab → New Pull Request
3
Set up the PR
base: main (where it goes into)
compare: feature-chatbot-engine (your branch)
4
Write title & description
Title: Add chatbot question loop
Body: short description of what you did
5
Click Create Pull Request
Your colleague can now review, comment, approve, or request changes.

The review process

👁️
Read your code
Your colleague sees all your changes side-by-side
💬
Leave comments on specific lines
Click any line in the diff to add a comment
✅
Approve or request changes
Once approved: click Merge pull request → Confirm merge. Your code is now in main.

After merging, clean up

# Go back to main git checkout main # Pull the newly merged code git pull # Delete your old branch (it's been merged) git branch -d feature-chatbot-engine

Understanding Git Status at All Times

Run git status constantly. It tells you exactly where you are.

git status # Run this constantly. # It tells you exactly where you are.
git log --oneline # Shows your last 10 commits as a # clean list. Good way to see what # happened and who did what.
What you see What it means
nothing to commit Your work is saved and pushed ✅
Changes not staged You edited files but haven't done git add yet
Changes to be committed You've done git add, ready to commit
Your branch is ahead by 1 commit You committed but haven't pushed yet
Your branch is behind Someone pushed to GitHub — you need to pull
Part 9

Resolving Conflicts

A conflict happens when two people edit the same line of the same file. Git doesn't know which version to keep, so it asks you.

What a conflict looks like

<<<<<<< HEAD def start_chatbot(): print("Welcome to PharmaBot") ======= def start_chatbot(): print("PharmaBot Ready") >>>>>>> feature-chatbot-engine
The <<<< HEAD block is what's in main.
The ==== / >>>> block is what's in your branch.

How to fix it

1
Delete the conflict markers
Remove the <<<<, ====, and >>>> lines
2
Keep the version you want
Or combine both into something better. Save the file.
3
Stage, commit, and push
git add . git commit -m "resolved merge conflict" git push

How to prevent most conflicts

🗂️ Stick to your assigned files (see Team Split)
⬇️ Pull at the start of every session
💬 Communicate before touching a file the other person owns

Part 12

Common Mistakes & How to Fix Them

You will encounter at least one of these. Bookmark this section.

😬 "I forgot to pull before coding"
git pull --rebase

This brings in the latest changes and replays your commits on top. Usually works cleanly.

😬 "I'm on main and I accidentally coded there"
# Create a branch to save your work git checkout -b my-accidental-work # Push it git push -u origin my-accidental-work # Go back to clean main git checkout main git reset --hard origin/main

Your work is safe on the new branch. Main is clean again.

😬 "My push was rejected"
! [rejected] main -> main (fetch first)

Means someone pushed since you last pulled. Fix:

git pull --rebase git push
😬 "I committed the wrong message"

If you haven't pushed yet:

git commit --amend -m "correct message here"
If you already pushed, just move on. Don't rewrite history on a shared repo.
😬 "I want to undo my last commit but keep the changes"
git reset --soft HEAD~1

Your changes stay, the commit disappears. You can re-commit with a better message.

  • fix
  • changes
  • stuff
  • asdfgh
✅ Good
  • add chatbot question loop in chatbot.py
  • fix tree_loader failing when JSON file is missing
  • write J001 euclid jam full decision tree
  • add sensor error option to main menu
Formula: verb + what + where (if not obvious)

Common verbs: add fix update remove rename refactor test
